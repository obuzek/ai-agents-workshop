# Synthetic EHR Patient Data

This directory contains 12 synthetic Electronic Health Record (EHR) patient files for the AI Agents Workshop labs. The data simulates a small GP practice — **Lakeview Family Medicine** — with three providers and a year's worth of patient portal messages.

## Files

| File | Purpose |
|---|---|
| `patients/patient_001.json` – `patient_012.json` | The 12 patient records |
| `schema.json` | JSON Schema (2020-12) that all patient files conform to |
| `patient_specs.json` | Diversity matrix and per-patient specifications used during generation |
| `validate_patients.py` | Validation script for structure, diversity, coherence, and voice checks |

## How this data was generated

### The problem with naive LLM generation

Asking an LLM to "generate 12 patient records" produces homogeneous data: similar ages, similar conditions, similar message styles, textbook-perfect lab values, and stereotypical demographic-to-attribute mappings. For realistic workshop labs, we needed data that was diverse, clinically coherent, and messy in the ways real patient data is messy.

### Our process

We used Claude as a **meta-agent** — an orchestrator that planned the generation, dispatched isolated workers, and validated the results. The entire pipeline ran in a single Claude Code session.

#### Step 1: Research and schema design

We started by researching real EHR structure, drawing from HL7 FHIR R4 resources (Patient, Condition, MedicationRequest, Observation, Encounter, Communication). We simplified the FHIR model into a flat JSON structure suitable for workshop use, but kept key design choices:

- **Coded concepts** with `system`, `code`, and `display` (using real ICD-10, LOINC, RxNorm, and CVX codes)
- **SOAP notes** (Subjective/Objective/Assessment/Plan) for encounter documentation
- **Communication-style messages** with sender/recipient roles, categories, priorities, and threaded replies
- **Lab results** with reference ranges and interpretation flags

The result is `schema.json`.

#### Step 2: Diversity matrix and patient specs

Before generating any patient data, we defined a strict diversity matrix in `patient_specs.json`. This is the most important step — it prevents the homogeneity problem by constraining every patient's profile before generation begins.

Each patient was assigned:
- **Demographics**: name, age, gender, occupation, insurance type, health literacy level
- **Clinical archetype**: a unique narrative (e.g., "polypharmacy elder," "diagnostic mystery," "silent until crisis")
- **Voice profile**: how the patient writes portal messages (formal, terse/SMS, anxious, rambling, clinical, humorous)
- **Message scenarios**: 3–6 specific portal message topics per patient, pre-assigned by category

We deliberately **decoupled demographics from clinical attributes** — names and implied ethnicity do not predict occupation, condition, or communication style. This counteracts a well-documented LLM bias pattern where, for example, an Indian name defaults to a software engineer or a Black name defaults to a construction worker.

#### Step 3: Isolated subagent generation

Each patient was generated by a **dedicated Claude subagent** — a separate context window that received only:
1. The JSON schema
2. That patient's spec (not the other 11)
3. Instructions to use real medical codes and maintain clinical coherence

This isolation prevents "cross-patient bleed," where an LLM reuses patterns from recently generated patients. We ran subagents in parallel batches of 4 (three batches total).

#### Step 4: Automated validation

After all 12 files were generated, we ran `validate_patients.py`, which checks:

- **Structural validity**: required fields present, correct types, valid coded concepts
- **Date ordering**: encounters and labs are chronologically consistent
- **Diversity**: age range spans 50+ years, gender distribution matches specs, no duplicate names, message categories are distributed
- **Clinical coherence**: diabetes patients have A1C labs and diabetes medications, hypertension patients have elevated BP readings, no medication-allergy conflicts, lab reference ranges are medically plausible
- **Voice differentiation**: message lengths vary across patients, informal markers present in low-literacy patients, at least one patient has messages written by a family member

#### Step 5: Clinical coherence review

We ran a second round of review using dedicated review agents (one per batch of patients), specifically checking:

- ICD-10, LOINC, RxNorm, and CVX codes are real and correctly applied
- Lab values are internally consistent (e.g., lipid panels follow the Friedewald equation: LDL = TC - HDL - TG/5)
- Medications match conditions at appropriate doses
- Encounter narratives are consistent with the patient's clinical trajectory

## Running validation

```bash
python data/validate_patients.py
```

This prints a colored report with per-patient summaries and pass/fail for each check category.

## Date ranges

- **Encounters and labs**: April 2023 – April 2026 (~3 years of history)
- **Patient portal messages**: April 2025 – April 2026 (last year only)

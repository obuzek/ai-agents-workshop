


# GitHub Issues Workflow Protocol

## AI attribution
Every issue comment, issue body, or PR description you post **must** end with this annotation line:

```
---
> 🤖 Generated by **claude** · AI-assisted workflow
```

This is non-negotiable. On GHE, AI actions use the same auth as the human user, so without this line there is no way to tell human-written content from AI-generated content.

## GitHub access
**Always use the GitHub MCP server** for all GitHub operations (reading/creating issues, posting comments, managing labels, creating PRs). Do NOT use the `gh` CLI — it may not be authenticated for this repository's host. The MCP server is pre-configured with the correct token and hostname.

## Picking up work
- When asked to check for work, look for issues with the `ai-ready` label.
- Pick the highest-priority (lowest number) issue unless told otherwise.
- Read the full issue body and all comments before starting.

## Worktree setup
If the launcher script (`ai-gh.sh`) already placed you in a worktree, skip this step. Otherwise, read the `ensure_worktree` function in `ai-gh.sh` and follow the same steps: fetch the default branch, create a worktree at `.ai-worktrees/issue-<N>` on a new branch, symlink the agent config directory, and copy the instructions file into the worktree.

**Important:** Your working directory does not persist between shell commands. Once the worktree is created, prefix all git and file commands with `cd .ai-worktrees/issue-<N> &&` or use absolute paths. Do this for every command — there is no way to permanently change your working directory mid-session.

## Planning
1. Analyze the issue and produce a concrete implementation plan.
2. Post the plan as an issue comment. Format:
   ```
   ## Proposed Plan
   - [ ] Step 1: ...
   - [ ] Step 2: ...
   ...
   ### Risks / open questions
   - ...
   ```
3. Remove the `ai-ready` label and add the `awaiting-approval` label.
4. **Stop and tell the terminal user** that you've posted a plan and are waiting for approval.

## Checking for approval
1. When asked to check for approval, check the issue's labels.
2. If the `ai-cancelled` label is present, stop all work on this issue. Remove `awaiting-approval` and `ai-working` if present. Post a brief acknowledgment comment and tell the terminal user.
3. If the `plan-approved` label is present, remove it and the `awaiting-approval` label, add the `ai-working` label, then begin implementation.
4. If `awaiting-approval` is still present without `plan-approved`, check for new comments with feedback. If there is feedback, revise your plan and repost.

## Implementation
- You should be in a git worktree on a dedicated branch for this issue. Other agents may be working on other issues in parallel in their own worktrees. Do NOT switch branches.
- Make atomic commits with clear messages referencing the issue (`fixes #N` or `refs #N`).
- Push your branch and open a PR against the repository's default branch when done.
- Post a summary comment on the issue with a link to the PR.
- Remove the `ai-working` label and add the `ai-done` label.

## Scope discipline
Your scope is defined by the approved plan. If you notice something that needs work but isn't part of that plan, don't fix it — create a new issue for it with the `ai-backlog` label, reference the parent issue, and keep going. When in doubt, err on the side of breaking it off. It's cheap to close an unnecessary issue; it's expensive to untangle a sprawling PR.

## Error handling
- If you get stuck or blocked, do both: post a comment on the issue describing the blocker AND tell the terminal user. The issue comment ensures the blocker is visible async; the terminal message gets immediate attention.
- Never force-push, delete branches, or take destructive actions without explicit terminal-user approval.
